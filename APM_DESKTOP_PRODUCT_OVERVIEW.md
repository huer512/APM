# APM Desktop 产品说明

APM Desktop 是一个 Agent 工作流运行与管理工具。用户可以选择预设工作流，填写任务参数并启动运行；运行过程中可以实时查看阶段进度、Agent 输出、工具调用和日志。如果流程需要人工判断，用户可以 attach 到运行实例中，向 Agent 补充说明、纠正方向，并决定何时进入下一阶段。运行失败时，APM 会帮助用户定位失败发生在哪个阶段、哪个 Agent、哪个工具调用，从而快速排查问题。

## 目标用户

APM Desktop 面向需要使用、配置和管理 Agent 工作流的人。

- 工作流使用者：选择一个已有 Agent 工作流，填写任务参数，然后运行。
- 工作流配置人员：维护工作流、Prompt、执行环境和默认参数。
- 调试 / 管理人员：观察运行状态、查看日志、处理失败、进行人工介入。

APM Desktop 不是一个单纯的聊天软件，也不是面向普通消费者的一键问答工具。它更像是一个 Agent 工作流控制台。

## 从打开桌面端到成功运行工作流

一个典型用户路径如下：

1. 打开 APM Desktop。
2. 确认后台服务已启动。如果没有启动，点击启动。
3. 检查基础设置是否完整，例如 API Key、工作目录和连接状态。
4. 进入仪表盘或运行页面。
5. 从列表中选择一个已有的工作流。
6. 填写本次运行需要的参数，例如任务目标、输入说明、项目路径等。
7. 点击运行。
8. 进入运行详情页。
9. 查看当前运行状态，包括正在执行哪个阶段、哪个 Agent、是否有工具调用、是否等待人工确认。
10. 等待工作流完成。
11. 查看最终输出、运行日志和每个阶段的执行结果。

可以概括为：

> 选择工作流，填写任务，点击运行，实时观察执行过程，必要时人工介入，最后查看结果。

## 运行失败后的排查路径

运行失败后，用户应该从高层到细节逐步排查。

1. 先看失败摘要。

   运行详情页应该明确显示失败原因、失败阶段、失败的 Agent，以及失败时间。

2. 查看失败阶段。

   确认工作流失败发生在哪个阶段，是刚启动就失败，还是中间某个阶段失败。

3. 查看 Agent 输出。

   查看失败前最后一个 Agent 收到的输入、产生的输出，以及是否有异常中断。

4. 查看工具调用。

   如果 Agent 调用了工具，需要查看工具名称、调用参数、返回结果和错误信息。很多问题会出现在文件访问、命令执行、网络请求、权限或路径配置上。

5. 查看日志级别。

   普通用户优先看错误和警告；调试人员再展开完整日志。

6. 检查配置。

   常见配置问题包括 API Key 缺失、工作流参数未填、执行环境不可用、工作目录不存在、远程主机不可达、权限不足。

7. 重试。

   如果是临时错误，可以重新运行；如果是某个阶段或 Agent 的问题，应先修正输入或配置后再运行。

可以概括为：

> APM Desktop 会把失败定位到具体阶段、Agent 和工具调用，用户可以从失败摘要、Agent 消息、工具事件和日志逐层排查。

## Attach 人工介入

Attach 是人工介入模式。它的定位不是接管整个系统，而是在工作流运行过程中让人参与决策、纠偏和继续推进。

用户 attach 进去后可以：

- 查看当前运行到哪个阶段。
- 查看当前哪个 Agent 正在执行。
- 查看该 Agent 的消息历史。
- 查看工具调用过程和结果。
- 向当前 Agent 发送补充说明。
- 纠正 Agent 的理解或方向。
- 提供缺失的信息。
- 在阶段完成后决定是否进入下一阶段。
- 暂时停留在当前阶段继续人工沟通。
- 退出人工介入，让流程继续按原工作流执行。

典型场景包括：

- Agent 理解任务有偏差，用户补充说明。
- 工作流到达关键节点，需要人工确认后继续。
- 工具调用失败，用户提供新的路径、凭据或替代方案。
- 中间结果不满意，用户要求当前 Agent 调整输出。
- 多阶段流程中，用户希望看完当前阶段结果再决定是否进入下一阶段。

可以概括为：

> Attach 让用户在工作流运行中插入人工判断：查看上下文、补充指令、纠正方向，并控制是否继续进入下一阶段。

## 工作流开发与配置管理

APM Desktop 不只用于运行工作流，也应该支持工作流的开发、维护和调试。对工作流配置人员来说，它是一个轻量的 Agent 工作流工作室。

工作流开发的核心目标是：

- 把一个复杂任务拆成多个阶段。
- 为每个阶段配置合适的 Agent 指令。
- 定义阶段之间的先后关系。
- 配置运行时需要的参数和执行环境。
- 在正式运行前检查配置是否完整、引用是否正确。
- 通过测试运行不断调整 Prompt 和流程结构。

一个典型的工作流开发路径如下：

1. 新建一个工作流。
2. 填写工作流名称、说明和默认参数。
3. 添加第一个阶段。
4. 为阶段选择或创建 Agent 指令。
5. 设置阶段完成后的下一步。
6. 继续添加后续阶段，形成完整流程。
7. 配置工作流的默认执行环境。
8. 检查参数、阶段、Agent 指令和执行环境是否完整。
9. 运行一次测试。
10. 根据运行结果、日志和 Agent 输出调整配置。
11. 保存为可复用的工作流。

对用户来说，Markdown 配置不应该只是“编辑文件”，而应该被理解为一种可版本化、可审查、可复用的工作流定义方式。APM Desktop 可以在保留 Markdown 灵活性的同时，提供更友好的管理能力：

- 浏览所有工作流、阶段、Agent 指令和执行环境。
- 新建、复制、重命名和删除配置。
- 编辑工作流说明、参数和 Prompt 内容。
- 检查缺失引用、无效参数、断开的阶段关系。
- 预览工作流结构。
- 对单个 Agent 指令进行测试。
- 对单个阶段进行测试。
- 对完整工作流进行测试运行。
- 比较不同版本的配置变更。
- 将成熟配置沉淀为团队模板。

可以概括为：

> 工作流开发是把 Agent 能力组织成稳定流程的过程。APM Desktop 应该让配置人员能够创建、检查、测试和维护这些流程，而不只是手工编辑一组 Markdown 文件。
